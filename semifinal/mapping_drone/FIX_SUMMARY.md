- [HIGH] mapping_drone/controller.py :: MappingController._register_sighting: The controller calls `self.run_writer.save_marker_image(frame.color_bgr, aruco_id, self._sighting_seq)` without forwarding the bbox. The run_writer's `save_marker_image` was extended to accept an optional `bbox_xyxy` parameter and draw a green rectangle + ArUco id label when given, but the caller never passes it. Net effect: every saved JPG is still a raw frame with no visible indication of which marker is in it. The README still promises 'bbox-overlaid frames of each sighting', so the prior HIGH-severity integration gap from the previous review is only half-fixed (writer supports it, controller doesn't use it).
  -> fix: Pass `bbox_xyxy=bbox` in the call: `img_path = self.run_writer.save_marker_image(frame.color_bgr, aruco_id, self._sighting_seq, bbox_xyxy=bbox)`.
- [HIGH] mapping_drone/run_writer.py :: RunWriter._sighting_to_dict + RunWriter.add_sighting: Regression introduced by the 'don't fabricate zeros' fix. `_sighting_to_dict` now OMITS required fields like `aruco_id` and `first_seen_at` from the returned dict when they are missing/non-coercible (only logs a warning). But `add_sighting` then does `pad_id = rec["aruco_id"]` (square-bracket access) on the very next line, which will raise KeyError and abort the sighting recording entirely. Previous behaviour silently substituted zeros; new behaviour crashes the writer on the first malformed sighting. Either failure mode is unacceptable, but the regression is that what used to be a soft-fail with bad data is now a hard-fail that loses ALL subsequent sightings (the exception bubbles up to the controller's `try/except` in `_register_sighting`, which only logs and continues, but the `_sightings` list / `_pads` dict were not updated and the persisted landing_pads.json never gets the new pad).
  -> fix: Decide explicitly: either keep skipping the field AND make `add_sighting` use `rec.get('aruco_id')` with an early-return-on-None guard, or raise a clear ValueError from `_sighting_to_dict` so callers see the contract violation. Recommend the early-return-on-None guard so a single malformed sighting doesn't poison the rest of the run.
- [MEDIUM] mapping_drone/controller.py :: MappingController._await_health: `_await_health` only checks the wall-clock deadline INSIDE the `async for` loop body, which only runs when the health stream yields. If the real MAVSDK health stream stalls (e.g. broken serial link), the function will block past the intended HEALTH_TIMEOUT_S and never return. Additionally, the `await asyncio.sleep(0.2)` inside the `async for` is meaningless — the stream's own pacing already drives iteration cadence; the sleep just adds latency between samples without providing the intended timeout safety.
  -> fix: Wrap the loop with `asyncio.wait_for(...)` and a small inner helper coroutine, OR start a `time.monotonic() > deadline` poll task in parallel and `asyncio.wait(..., return_when=FIRST_COMPLETED)`. Remove the inner `await asyncio.sleep(0.2)`.

The pre-existing issue from the prior review (`is_local_position_ok` not checked here) was addressed by moving the health check into `run_mission._await_health`, but the connection-state wait itself still has no bound.

- [LOW] mapping_drone/controller.py :: _load_waypoints: `data = json.loads(Path(path).read_text())` followed by `for row in data` blindly assumes the JSON top-level is a list of 3-element rows. If a user passes `{"waypoints": [...]}` (a common pattern they might infer from the project layout) the iteration silently yields the dict's keys (strings), and `len(row) != 3` raises a misleading error 'waypoint must be [n, e, alt], got <key-name>'. The error path also doesn't validate that altitude is positive or that the waypoint set is reachable.
  -> fix: Validate `isinstance(data, list)` first with a clear error, and add at least a positive-altitude sanity check on each row.
- [LOW] mapping_drone/controller.py :: MappingController._check_safety stuck-watchdog: The position-stuck watchdog only considers N/E movement (`moved = math.hypot(ds, de)`) but the first waypoint is `(0.0, 0.0, 1.5)` which is the same N/E as the takeoff origin — only altitude changes. If the drone ascends in place for the entire STUCK_WINDOW_S (20 s), the watchdog will erroneously fire 'position-stuck' even though the drone is making perfect progress (climbing). With the airborne-only grace of STUCK_GRACE_S=10s + STUCK_WINDOW_S=20s, this triggers only on slow climbs, but the failure mode is silent: the mission aborts and the operator sees 'position-stuck' instead of 'waypoint unreachable'.
  -> fix: Include altitude in the movement metric: `moved = math.sqrt(ds**2 + de**2 + dd**2)` where `dd = max(downs) - min(downs)`. Or skip the watchdog while the drone is still in the climb-to-first-waypoint phase by requiring `>2 waypoints reached` first.
- [LOW] mapping_drone/run_writer.py :: STATUS.txt key acceptance: The fix added `_pad_is_valid` to accept either `validity_classification` or `valid`, and the controller's `_unique_pads` emits dicts with `valid`. This works, but the now-two-shapes-accepted contract is fragile: future contributors writing new producers may add a third spelling. The codebase has three different validity key names floating around (`valid`, `validity_classification`, and the in-memory pad dict's `valid` field which is the controller-facing name). Pick one.
  -> fix: Make the controller's `_unique_pads` emit `validity_classification` directly, then drop the legacy `valid` fallback from `_pad_is_valid` (or vice versa).
- [LOW] mapping_drone/controller.py :: MockMavsdk._tick called from multiple sources: `_tick` is invoked from `position_velocity_ned`, `attitude_euler`, and `set_velocity_ned`, each at different rates. Because all three reset `self._last_update` to `now`, whichever was last computes the dt and the others see dt~=0. Net effect on integration: the mock position drifts at slightly less than the commanded velocity because attitude/position pumps consume the dt slices between offboard ticks. Not a correctness bug, but the mock-drone trajectory will undershoot waypoints slightly and may interact with the stuck-watchdog in long mock runs.
  -> fix: Either centralise `_tick` to one source (e.g. only inside `position_velocity_ned`) or use separate timestamps per pump.
