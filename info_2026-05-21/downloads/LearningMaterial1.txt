TWIThaLtEis PX4?

SUBTITLE

 · PX4 is a quadcopter autopilot platform
     used in both the Qualifier and Final
     challenges.

 · The autopilot acts as the brain of the
     quadcopter, processing sensor data and
     controlling the motors to achieve stable
     flight.

 · The flight stack provides essential
     stabilization and safety features, as well
     as a level of pilot assistance for both
     manual and autonomous flight.

TMIiTsLsiEon Computer

SUBTITLE

 · Mission Computer or companion computer
     performs high-level intelligence and then
     control the PX4.

 · The PX4 passes the control "offboard" to the
     Mission Computer.

 · The PX4 and Mission Computer communicate
     over a high-speed serial or IP link using the
     MAVLink protocol.

 · Most Mission Computers run a Linux-based OS
     such as Ubuntu 22.04.

 · The challenges require teams to develop the
     control logic that runs on the Mission
     Computer.

 · During the Qualifier, the Mission Computer will
     be your code running on either the team's
     laptop or the organiser's laptop.

TMIATVLSEDK

SUBTITLE

 · MAVLink (Micro Air Vehicle Link) is a
     lightweight, header-only messaging protocol
     designed for bandwidth-constrained radio
     links to communicate between unmanned
     vehicles (drones) and ground control stations
     (GCS) or internal components.

 · MAVSDK (or MAV Software Development Kit) is
     a set of high-level, cross-platform libraries
     that take cares of sending MAVLink-
     instructions to MAVLink supported drones and
     systems (like PX4).

 · MAVSDK supports languages including C++,
     Python, Swift, and Java.

 · Your team mainly focuses on developing the
     code in the mission computer for the high level
     autonomy logic.

 · The workshop focuses mostly on python and
     sometimes C++.

TSIimTLuElator

SUBTITLE

· The Qualifier challenge and workshop uses
     simulation environment. "World" is simulated
     using Gazebo Harmonic.

· The simulation environment is in the VMWare
     virtual desktop machine, and runs within the
     virtual machine.

· PX4 also runs within the virtual machine in
     Software-In-The-Loop mode (SITL PX4), to
     emulate a real-life drone in the simulation
     environment.

· The Gazebo gives SITL PX4 the simulated
     sensors data, and also provides camera or
     depth camera feeds to your code.

· Your code sends commands to PX4 SITL,
     which then controls the virtual drone in the
     simulation.

· Ground Control Station, QGroundControl,
     monitors and controls the emulated PX4.
      · QGroundControl has been installed into
           the VMWare virtual desktop machine.

CTIoTnLsEiderations

SUBTITLE

· Graceful Exit: Pressing Ctrl+C will raise
     KeyboardInterrupt. In production code,
     wrap asyncio.run(run()) in a try/except
     or use signal handlers to send a safe
     drone.action.land() or
     drone.action.kill() before exiting.

· SITL vs Real Hardware: This code works
     identically on a real drone, but you must
     change system_address to
     "serial:///dev/ttyACM0:57600" if the
     your code runs on single board computer that
     connects to the PX4 flight controller via serial
     port or UDP/TCP address if the drone has
     remote access wifi.

TReITfeLrEto basic_offboard_control.py                                                                                                                        23

SUBTITLE

· Autonomy capabilities are achieved using
     companion computer that performs autonomous
     logic, instead of the PX4 flight control.

· Then the companion computer then control PX4 via
     offboard mode. PX4 handles the low-level motor
     controls based on the high level instructions from
     companion computer.

· Offboard mode means that the flight controller
     relinquish autonomous flight logics to an external
     computer via MAVLINK. Key concepts when using
     offboard mode:

· Start by calling offboard.start() and end use
     offboard.stop().

· PX4 requires offboard setpoints at least every 0.5
     seconds. Meaning your code must send a setpoint
     command every 0.5 seconds (notice the
     asyncio.sleep (0.2)). If not, PX4 will switch
     back control to the next flight mode as a form of
     failsafe.

· The code demonstrates controlling the PX4 using
     set velocity setpoint command - await
     drone.offboard.set_velocity_ned(vel)

· Important concept for velocity: The Coordinate
     frame of PX4 is North (+X), East (+y) and Down (+z)
     Yaw is clockwise from north. -ve for z means up.

TKeITyLcEoncepts in Fly_to_coordinate.py  [NSTRUCTOR TO DEMO AND EXPLAIN
                                          FLY_TO_COODINATE.PY]
SUBTITLE

· drone.action.goto_location(lat, lon,
     alt, yaw): High-level PX4 command.
     Automatically switches to HOLD  GOTO mode and
     uses PX4's internal position controller.

· NED vs GPS: This example uses Global Coordinates
     (Lat/Lon/Alt MSL). PX4 internally converts to NED
     relative to home.

· Arrival Detection: MAVSDK doesn't provide a native
     "arrived" callback. You must poll
     drone.telemetry.position() and compute
     distance. The Haversine formula handles Earth
     curvature accurately for drone-scale distances.

· Tolerance & Timeout: Real systems oscillate.
     pos_tolerance=1.0m and
     alt_tolerance=0.5m prevent infinite waiting.
     Timeout prevents script hang on PX4 navigation
     failures.

· Real Hardware Warning: On physical drones, always:
· Test in COM_RC_IN_MODE=1 (Offboard/MAVLink

     allowed)

· Keep RC sticks ready for manual override, especially
     switch out of offboard mode into hold mode when
     something went wrong.

· Set COM_OBL_ACT=1 (Land) or 2 (Hold) for
     offboard failures
